<objective>
Build the complete API key lifecycle: tRPC router for creating project-scoped keys (via Better Auth metadata pattern), listing keys per project, and revoking keys, plus the dashboard UI with Stripe-like key management UX (show once, copy button, labeled keys list, revoke action). Also create the `verifyProjectKey` utility that Phase 2 will use for API authentication.

Purpose: API keys are the authentication mechanism for the public REST API (Phase 2). Without keys, external developers cannot submit entries. The `verifyProjectKey` utility isolates the metadata-based project scoping logic in one place — Phase 2's middleware calls this utility rather than implementing raw Better Auth verification with ad-hoc metadata checks.

Output: Working API key management — user can generate keys with labels, copy the key once, see their keys listed, and revoke keys. A shared `verifyProjectKey` utility is ready for Phase 2's API middleware.
</objective>

  <action>
    **Step 1: API key tRPC router**

    Create `packages/trpc/src/routers/api-key.ts` with the following procedures:

    **apiKey.create** — protectedProcedure mutation
    - Input: `z.object({ projectId: z.string().uuid(), label: z.string().max(50).optional() })`
    - Step 1: Verify project ownership — query projects table with `and(eq(projects.id, input.projectId), eq(projects.userId, ctx.user.id))`. Throw `NOT_FOUND` if missing.
    - Step 2: Call `ctx.auth.api.createApiKey({ body: { name: input.label ?? 'API Key', prefix: 'ph_live_', metadata: { projectId: input.projectId } }, headers: ctx.requestHeaders })`.
    - NOTE on metadata: Better Auth may expect `metadata` as a JSON string or object depending on version. Check Better Auth docs — if it expects string, use `JSON.stringify({ projectId: input.projectId })`. If object, pass directly. Try object first, fall back to stringify if errors occur.
    - Step 3: Return `{ id: result.id, key: result.key, start: result.start, name: result.name }`.
    - CRITICAL: `result.key` is the FULL PLAINTEXT key — this is the ONLY time it's available. Return it to the client for one-time display. After this mutation response, the key is gone forever.

    **apiKey.list** — protectedProcedure query
    - Input: `z.object({ projectId: z.string().uuid() })`
    - Step 1: Verify project ownership (same pattern as create).
    - Step 2: Query apikeys table for keys belonging to the current user where metadata contains the projectId.
      - Option A (preferred): Query `apikeys` where `userId = ctx.user.id`, then filter in JS where `JSON.parse(metadata)?.projectId === input.projectId`.
      - Option B: Use Drizzle SQL template for JSON extraction if supported.
      - Use Option A — it's simpler and key counts are small enough for in-memory filtering.
    - Step 3: Return array of `{ id, name, start, prefix, enabled, createdAt }` — NEVER return the `key` (hash) column.
    - Sort by createdAt descending.

    **apiKey.revoke** — protectedProcedure mutation
    - Input: `z.object({ keyId: z.string() })`
    - Step 1: Look up the key to get its metadata → extract projectId → verify project ownership. If the key doesn't belong to a project owned by this user, throw `NOT_FOUND`.
    - Step 2: Call `ctx.auth.api.revokeApiKey({ body: { keyId: input.keyId }, headers: ctx.requestHeaders })`.
    - Step 3: Return `{ success: true }`.
    - KEYS-03: Revoked keys immediately reject subsequent API calls (Better Auth sets `enabled: false`).

    **Update router.ts:**
    - Import `apiKeyRouter` from `./routers/api-key`.
    - Add to appRouter: `apiKey: apiKeyRouter`.

    **Step 2: verifyProjectKey utility**

    Create `packages/auth/src/verify-project-key.ts`:
    - Export `verifyProjectKey(auth, key: string, projectId: string): Promise<{ valid: boolean; userId?: string; keyId?: string }>`.
    - Step 1: Call Better Auth's key verification. This may be `auth.api.verifyApiKey({ body: { key } })` or direct DB lookup — check Better Auth docs for server-side verification.
    - If Better Auth doesn't expose a clean server-side verify, fall back to:
      1. Hash the key using the same algorithm Better Auth uses.
      2. Look up in apikeys table where hash matches and enabled is true.
      3. Parse metadata and check projectId.
    - Step 2: If key is invalid, expired, or disabled → return `{ valid: false }`.
    - Step 3: Parse `result.metadata` (JSON string) → extract `projectId`.
    - Step 4: If `metadata.projectId !== projectId` → return `{ valid: false }` (key exists but wrong project).
    - Step 5: Return `{ valid: true, userId: result.userId, keyId: result.id }`.
    - CRITICAL (Pitfall 1 from RESEARCH.md): This utility MUST check metadata.projectId, not just key existence. Without this check, any valid key from any project could access any other project's data.
    - This utility is consumed by Phase 2's API middleware. Isolating it here means if Better Auth ships native project scoping (issue #4746), we swap internals here without touching callers.

    Update `packages/auth/src/index.ts` to export `verifyProjectKey`.
