<!-- Key API contracts for QA verification -->

Auth signup endpoint (Better Auth):
POST /api/auth/sign-up/email
Body: { name: string, email: string, password: string }
Returns: { user: { id, name, email }, session: { id, token } }

Auth signin endpoint (Better Auth):
POST /api/auth/sign-in/email
Body: { email: string, password: string }
Returns: { user: {...}, session: { id, token } }

tRPC project.create:
Input: { name: string, mode: "waitlist" | "demo-booking" }
Returns: project object with id

tRPC project.updateFields:
Input: { projectId: string, collectName: boolean, collectCompany: boolean, collectMessage: boolean }
Returns: updated field config

Entry submission:
POST /api/v1/entries
Headers: { x-api-key: "ph_live_..." }
Body: { email: string, name?: string, company?: string, message?: string, metadata?: object }
Returns 201: { data: { id, email, name, company, position, createdAt } }
Returns 200: (duplicate) { data: { id, email, name, company, position, createdAt } }
Returns 400: { error: { code: "VALIDATION_ERROR", message, details } }
Returns 401: { error: { code: "MISSING_API_KEY" | "INVALID_API_KEY", message } }
Returns 429: Rate limit exceeded

Rate limit: 60 requests per minute per API key, IETF draft-6 headers
</interfaces>

**Prerequisites — Start the dev stack:**

2. Verify Redis is running (check with `redis-cli -p 6380 ping`)
3. Start the development server: `pnpm dev` from the repo root (this starts API on 3001 and web on 5173)
4. Wait for "API server running on http://localhost:3001" in the output

**QA-01: Full Developer Flow**

Step 1 — Sign up a new user:
```bash
curl -s -X POST http://localhost:3001/api/auth/sign-up/email \
  -H "Content-Type: application/json" \
  -d '{"name":"QA Tester","email":"qa-tester@pleasehold.dev","password":"SecurePass123!"}' | jq .
```
- Verify: Response contains `user.id`, `user.email === "qa-tester@pleasehold.dev"`, and a `session` object
- Record the session token from the `set-cookie` header for subsequent requests

Step 2 — Create a project:
```bash
# Extract session cookie from signup response, use it for tRPC call
curl -s -X POST http://localhost:3001/trpc/project.create \
  -H "Content-Type: application/json" \
  -H "Cookie: <session-cookie-from-step-1>" \
  -d '{"json":{"name":"QA Waitlist","mode":"waitlist"}}' | jq .
```
- Verify: Response contains `result.data.json` with project `id` and `name === "QA Waitlist"`
- Record the project ID

Step 3 — Configure fields (enable name collection):
```bash
curl -s -X POST http://localhost:3001/trpc/project.updateFields \
  -H "Content-Type: application/json" \
  -H "Cookie: <session-cookie>" \
  -d '{"json":{"projectId":"<project-id>","collectName":true,"collectCompany":false,"collectMessage":false}}' | jq .
```
- Verify: Response confirms `collectName: true`

Step 4 — Generate an API key:
```bash
curl -s -X POST http://localhost:3001/api/auth/api-key/create \
  -H "Content-Type: application/json" \
  -H "Cookie: <session-cookie>" \
  -d '{"name":"QA Key","metadata":{"projectId":"<project-id>"}}' | jq .
```
- Note: The exact auth API path for key creation may differ — check Better Auth docs. It might be via tRPC `apiKey.create` instead:
```bash
curl -s -X POST http://localhost:3001/trpc/apiKey.create \
  -H "Content-Type: application/json" \
  -H "Cookie: <session-cookie>" \
  -d '{"json":{"projectId":"<project-id>","name":"QA Key"}}' | jq .
```
- Verify: Response contains the full API key string starting with `ph_live_`
- Record the API key for subsequent entry submissions

Step 5 — Submit an entry via curl:
```bash
curl -s -X POST http://localhost:3001/api/v1/entries \
  -H "Content-Type: application/json" \
  -H "x-api-key: <api-key>" \
  -d '{"email":"first-entry@example.com","name":"QA First Entry"}' | jq .
```
- Verify: Status 201, response contains `data.id`, `data.email === "first-entry@example.com"`, `data.position >= 1`, `data.createdAt` is ISO string

Step 6 — Submit a second entry to verify sequential positioning:
```bash
curl -s -X POST http://localhost:3001/api/v1/entries \
  -H "Content-Type: application/json" \
  -H "x-api-key: <api-key>" \
  -d '{"email":"second-entry@example.com","name":"QA Second Entry"}' | jq .
```
- Verify: Status 201, position is one more than the first entry's position

**QA-05: API Key Security**

Step 7 — Submit without API key (expect 401):
```bash
curl -s -w "\nHTTP_STATUS:%{http_code}\n" -X POST http://localhost:3001/api/v1/entries \
  -H "Content-Type: application/json" \
  -d '{"email":"no-key@example.com"}' | jq .
```
- Verify: Status 401, `error.code === "MISSING_API_KEY"`

Step 8 — Submit with invalid API key (expect 401):
```bash
curl -s -w "\nHTTP_STATUS:%{http_code}\n" -X POST http://localhost:3001/api/v1/entries \
  -H "Content-Type: application/json" \
  -H "x-api-key: ph_live_invalid_key_does_not_exist_12345" \
  -d '{"email":"bad-key@example.com"}' | jq .
```
- Verify: Status 401, `error.code === "INVALID_API_KEY"`

Step 9 — Verify rate limiting triggers at threshold:
```bash
# Send 65 rapid requests (limit is 60/min) and check for 429
for i in $(seq 1 65); do
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:3001/api/v1/entries \
    -H "Content-Type: application/json" \
    -H "x-api-key: <api-key>" \
    -d "{\"email\":\"rate-limit-${i}@example.com\"}")
  echo "Request $i: HTTP $STATUS"
done
```
- Verify: First 60 requests return 201 (or 200 for dedup), requests 61+ return 429
- Note the exact request number where 429 first appears

Step 10 — Verify valid key inserts entries into database:
```bash
# Query the database directly to confirm entries exist
# Use the DATABASE_URL from the local .env or default
psql "$DATABASE_URL" -c "SELECT id, email, position, status FROM entries WHERE project_id = '<project-id>' ORDER BY position LIMIT 10;"
```
- Verify: Entries from steps 5-6 and rate limit test appear with sequential positions
