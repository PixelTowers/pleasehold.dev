<interfaces>
<!-- Notification channel configuration contracts from tRPC router -->

tRPC notification.create:
Input varies by channel type:
- email: { projectId, type: "email", name, config: { recipientEmail: string } }
- slack: { projectId, type: "slack", name, config: { webhookUrl: string } }
- discord: { projectId, type: "discord", name, config: { webhookUrl: string } }
- telegram: { projectId, type: "telegram", name, config: { botToken: string, chatId: string } }
- webhook: { projectId, type: "webhook", name, config: { url: string } }
  (webhook secret is auto-generated on create)

tRPC project.toggleDoubleOptIn:
Input: { projectId: string, enabled: boolean }
Returns: updated project with doubleOptIn flag

Verification endpoint:
GET /verify/{token}
Returns 200: { data: { verified: true, email: string } }
Returns 400: { error: { code: "INVALID_TOKEN", message } }

Notification job types:
- "entry_created" — dispatched to all channels when entry is submitted (no double opt-in) or after verification
- "verification_email" — sends verification email when double opt-in is enabled

Worker channel senders:
- email: uses Nodemailer SMTP transport
- slack: POST to webhook URL with JSON payload
- discord: POST to webhook URL with JSON payload
- telegram: POST to api.telegram.org/bot{token}/sendMessage
- webhook: POST to URL with JSON body + X-PleaseHold-Signature header (HMAC-SHA256)
</interfaces>

Test the complete double opt-in flow: enable → submit → verification email → click verify link → status flips → notification fires (QA-03).

**Step 1 — Enable double opt-in on the project:**
```bash

  -d '{"json":{"projectId":"<PROJECT_ID>","enabled":true}}' | jq .
```
- Verify: Response confirms `doubleOptIn: true`

**Step 2 — Submit an entry (should trigger verification email, NOT notification):**
```bash
ENTRY_RESPONSE=$(curl -s -X POST http://localhost:3001/api/v1/entries \
  -H "Content-Type: application/json" \
  -H "x-api-key: <API_KEY>" \
  -d '{"email":"verify-test@example.com"}')
echo "$ENTRY_RESPONSE" | jq .
ENTRY_ID=$(echo "$ENTRY_RESPONSE" | jq -r '.data.id')
```
- Verify: Status 201, entry created

**Step 3 — Verify entry status is pending_verification:**
```bash
# Query the database directly
psql "$DATABASE_URL" -c "SELECT id, email, status, verification_token, verification_expires_at FROM entries WHERE id = '$ENTRY_ID';"
```
- Verify: status = 'pending_verification', verification_token is a UUID, verification_expires_at is ~48 hours in the future

**Step 4 — Check that a verification EMAIL was sent (not a notification):**
Check Mailpit:
```bash
curl -s http://localhost:8025/api/v1/messages | jq '.messages[0]'
```
- Verify: Email was sent to verify-test@example.com
- Verify: Email contains a verification link with the token (format: /verify/{uuid})
- Extract the verification token from the email body or from the DB query in step 3

Check webhook capture server:
- Verify: NO notification webhooks were sent for this entry (double opt-in means notifications fire AFTER verification, not on submission)

**Step 5 — Click the verification link:**
```bash
# Use the verification token from the DB query
VERIFY_TOKEN=$(psql "$DATABASE_URL" -t -c "SELECT verification_token FROM entries WHERE id = '$ENTRY_ID';" | tr -d ' \n')
curl -s http://localhost:3001/verify/$VERIFY_TOKEN | jq .
```
- Verify: Status 200, response `{ data: { verified: true, email: "verify-test@example.com" } }`

**Step 6 — Verify entry status flipped to 'new':**
```bash
psql "$DATABASE_URL" -c "SELECT id, email, status, verified_at FROM entries WHERE id = '$ENTRY_ID';"
```
- Verify: status = 'new', verified_at is set to a timestamp

**Step 7 — Verify post-verification notifications fired:**
Wait 5-10 seconds for worker to process the notification job, then:

Check worker logs for "entry_created" job completion after verification.

Check webhook capture server for new POSTs (Slack, Discord, webhook should all receive).

Check Mailpit for a new notification email (separate from the verification email):
