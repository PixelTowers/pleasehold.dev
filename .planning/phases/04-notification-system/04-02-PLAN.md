    - Send to all recipients with subject "New entry on {projectName}: {email}"

    - Export `sendSlackNotification(webhookUrl: string, entry: EntryPayload): Promise<void>`
    - POST JSON to webhookUrl with `text` (fallback) and `blocks` array containing a section block with mrkdwn text
    - Use native `fetch()` -- no SDK. Format: `*New entry on {projectName}*\nEmail: {email}\nName: {name}\nPosition: #{position}`
    - Throw on non-ok response with status code in error message

    Create `apps/worker/src/senders/discord.ts`:
    - Export `sendDiscordNotification(webhookUrl: string, entry: EntryPayload): Promise<void>`
    - POST JSON to webhookUrl with `content` text and `embeds` array containing one embed with title "New Entry", fields for email/name/position, color 0x5865f2 (Discord blurple)
    - Use native `fetch()`. Throw on non-ok response.

    Create `apps/worker/src/senders/telegram.ts`:
    - Export `sendTelegramNotification(botToken: string, chatId: string, entry: EntryPayload): Promise<void>`
    - POST to `https://api.telegram.org/bot${botToken}/sendMessage` with chat_id, text (Markdown formatted), parse_mode: 'Markdown'
    - Use native `fetch()`. Throw on non-ok response, including response body in error for debugging.

    Create `apps/worker/src/senders/webhook.ts`:
    - Export `sendWebhookNotification(url: string, secret: string, payload: Record<string, unknown>): Promise<void>`
    - Import `crypto` from `node:crypto`
    - Compute HMAC-SHA256: `const timestamp = Math.floor(Date.now() / 1000).toString(); const signaturePayload = timestamp + '.' + body; const signature = crypto.createHmac('sha256', secret).update(signaturePayload).digest('hex');`
    - Send headers: `Content-Type: application/json`, `X-PleaseHold-Signature: signature`, `X-PleaseHold-Timestamp: timestamp`
    - Throw on non-ok response with status code

    Create `apps/worker/src/senders/verification-email.ts`:
    - Export `sendVerificationEmail(email: string, verificationToken: string, projectName: string): Promise<void>`
    - Use same Nodemailer transporter pattern as email.ts (or share a transporter module -- use a shared `apps/worker/src/senders/mailer.ts` that creates and exports the transporter singleton)
    - Build verification URL: `${process.env.API_URL}/api/v1/verify/${verificationToken}`
    - Send with subject "Confirm your submission to {projectName}"
    - HTML body with a styled "Confirm Submission" button linking to the verification URL
    - Plain text fallback with the URL
    - Use SMTP_FROM env var (default: 'noreply@pleasehold.dev')

    Create shared `apps/worker/src/senders/mailer.ts`:
    - Export a Nodemailer transporter singleton created from SMTP env vars
    - Log a warning at import time if SMTP_HOST is not set

    Define a shared `EntryPayload` type used by all senders. Create `apps/worker/src/types.ts`:
    - `export interface EntryPayload { email: string; name?: string | null; company?: string | null; position: number; projectName: string; }`

    Create `apps/worker/src/processor.ts`:
    - Export `processNotification` async function compatible with BullMQ Job processor signature
    - Import createDb from @pleasehold/db and create a DB connection (singleton, created once at module level)
    - For job type 'entry_created':
      1. Fetch the entry by entryId from DB (if not found, log warning and return -- entry may have been deleted)
      2. Fetch the project by projectId with name (for notification messages)
      3. Fetch all enabled notification channels for the project: `db.select().from(notificationChannels).where(and(eq(notificationChannels.projectId, data.projectId), eq(notificationChannels.enabled, true)))`
      4. For each channel, dispatch to the appropriate sender based on channel.type:
         - 'email': call sendEmailNotification with config.recipients (string[]), entry payload
         - 'slack': call sendSlackNotification with config.webhookUrl, entry payload
         - 'discord': call sendDiscordNotification with config.webhookUrl, entry payload
         - 'telegram': call sendTelegramNotification with config.botToken, config.chatId, entry payload
         - 'webhook': call sendWebhookNotification with config.url, config.secret, payload object containing entry data
      5. After each channel send attempt, insert a notification_logs row with status 'sent' or 'failed' (with error message)
      6. If ANY channel send fails, log the error but continue to the next channel (do not let one channel failure block others)
      7. After all channels attempted, if any failed, throw an error with summary (BullMQ will retry the entire job -- but the processor should skip channels that already have a 'sent' log for this entry to avoid duplicate notifications on retry)
    - For job type 'verification_email':
      1. Fetch the entry by entryId (must have verificationToken set)
      2. Fetch the project name
      3. Call sendVerificationEmail with entry.email, entry.verificationToken, project.name
      4. Log success
    - IMPORTANT: Read channel secrets (webhook HMAC secret) from the database at dispatch time, NOT from job data. Job data may be visible in monitoring tools.

    Modify `apps/worker/src/index.ts`:
    - Replace the placeholder processor with the real `processNotification` import from './processor'
    - Keep all existing startup logic (Redis check, graceful shutdown, event listeners)
