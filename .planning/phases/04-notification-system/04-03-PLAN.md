<objective>
Build the notification channel management tRPC router, the dashboard notification settings page, and the double opt-in project toggle.

Purpose: This is the user-facing layer that lets project owners configure which notification channels to use, manage their settings, and enable double opt-in. Without this, the notification pipeline built in Plan 04-02 has no channels to dispatch to.

Output: A notification tRPC router with full CRUD for channels, a notification settings page in the project dashboard, per-channel-type configuration forms, a double opt-in toggle on the project, and the entry submission route wired to conditionally apply double opt-in (pending_verification status + verification email) based on the project setting.
</objective>

    Procedures:

    **list** (query): List all notification channels for a project
    - Input: `{ projectId: z.string().uuid() }`
    - Verify project ownership
    - Return all channels for the project, ordered by createdAt asc
    - For webhook type channels, MASK the secret in the response (return only first 8 chars + '...'). The secret is shown once on creation only.

    **create** (mutation): Create a new notification channel
    - Input: `{ projectId: z.string().uuid(), type: z.enum(['email', 'slack', 'discord', 'telegram', 'webhook']), config: z.record(z.unknown()) }`
    - Verify project ownership
    - Validate config against the type-specific Zod schema (switch on type, parse config against appropriate schema)
    - For webhook type: auto-generate an HMAC secret using `crypto.randomBytes(32).toString('hex')` and merge into config as `secret`
    - Insert into notificationChannels table
    - Return the created channel. For webhook type, return the full secret THIS TIME ONLY (it will be masked in subsequent list calls).

    **update** (mutation): Update an existing channel's config or enabled status
    - Input: `{ projectId: z.string().uuid(), channelId: z.string().uuid(), enabled: z.boolean().optional(), config: z.record(z.unknown()).optional() }`
    - Verify project ownership
    - Fetch the channel, verify it belongs to the project
    - If config is provided, validate against the type-specific schema
    - For webhook type updates: do NOT allow changing the secret via update. If user provides a secret in config, strip it. The secret can only be regenerated (see regenerateSecret).
    - Update the channel with new config/enabled, set updatedAt to now()
    - Return updated channel (with masked webhook secret)

    **delete** (mutation): Delete a notification channel
    - Input: `{ projectId: z.string().uuid(), channelId: z.string().uuid() }`
    - Verify project ownership
    - Delete the channel where id AND projectId match
    - Return `{ deleted: true }`

    **regenerateSecret** (mutation): Generate a new HMAC secret for a webhook channel
    - Input: `{ projectId: z.string().uuid(), channelId: z.string().uuid() }`
    - Verify project ownership
    - Fetch channel, verify type === 'webhook'
    - Generate new secret: `crypto.randomBytes(32).toString('hex')`
    - Update config with new secret
    - Return the new secret (shown once)

    **toggleDoubleOptIn** (mutation): Toggle the doubleOptIn flag on a project
    - Input: `{ projectId: z.string().uuid(), enabled: z.boolean() }`
    - Verify project ownership
    - Update projects table: set doubleOptIn to input.enabled, updatedAt to now()
    - Return `{ doubleOptIn: input.enabled }`

    **getDoubleOptIn** (query): Get current doubleOptIn status
    - Input: `{ projectId: z.string().uuid() }`
    - Verify project ownership
    - Return `{ doubleOptIn: project.doubleOptIn }`
