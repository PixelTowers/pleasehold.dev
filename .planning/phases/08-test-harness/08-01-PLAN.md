  <action>
Create three files that form the test database harness:

**apps/api/vitest.config.ts** — Vitest configuration:
- Use `defineConfig` from `vitest/config`
- Set `test.globalSetup` to `./src/test/setup.ts`
- Set `test.testTimeout` to 15000 (DB queries can be slow in CI)
- Set `test.env` to load from process.env (no .env.test file magic — use DATABASE_URL directly)

**apps/api/src/test/setup.ts** — Global setup/teardown:
- Export `setup()` and `teardown()` functions (Vitest global setup contract)
- `setup()`:
  1. Read `DATABASE_URL` from env. If missing, throw with message: "DATABASE_URL required for integration tests. Point it at a test PostgreSQL instance."
  2. Derive a unique test database name: `pleasehold_test_{timestamp}` to avoid collisions
  3. Connect to the base PostgreSQL URL (replace DB name with `postgres`) using `postgres` library directly
  4. Run `CREATE DATABASE {test_db_name}`
  5. Connect to the new test DB using `createDb()` from `@pleasehold/db`
  6. Push schema using Drizzle's `migrate()` or by shelling out: `pnpm drizzle-kit push --force-accept-data-loss` with the test DATABASE_URL. Use whichever approach works — if `drizzle-kit push` is simpler (it reads schema from the package), use that via `execSync`.
  7. Run the seed function from `./seed.ts`
  8. Store the test DATABASE_URL in `process.env.TEST_DATABASE_URL` so tests can read it
  9. Also store `test_db_name` for teardown
- `teardown()`:
  1. Connect to base PostgreSQL URL
  2. Terminate connections to the test DB: `SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE datname = '{test_db_name}'`
  3. `DROP DATABASE {test_db_name}`

**apps/api/src/test/seed.ts** — Seed function:
- Export `async function seed(db: Database)` and also export the seeded constants (API key, project ID, etc.) so tests can import them
- Seed data must satisfy the API key auth middleware chain:
  1. Insert a user into Better Auth's `user` table (the `authUsers` table): `{ id: 'test-user-1', name: 'Test User', email: 'owner@test.com', emailVerified: true, createdAt, updatedAt }`
  2. Insert a project: `{ id: 'test-project-1', userId: 'test-user-1', name: 'Test Project', mode: 'waitlist', doubleOptIn: false }`
  3. Insert a field config: `{ projectId: 'test-project-1', collectName: false, collectCompany: false, collectMessage: false }` (email-only for simplicity)
  4. Insert an API key into Better Auth's `apiKey` table. The key must:
     - Have a known plaintext value that tests can send as `x-api-key` header
     - Have `metadata` containing `{ projectId: 'test-project-1' }` (this is how the middleware resolves the project)
     - Better Auth's `apiKey` table stores hashed keys. Research the table schema — if Better Auth hashes the key, you need to either: (a) use Better Auth's API to create the key programmatically, or (b) insert a row with the hash matching a known plaintext. Option (a) is preferred if feasible.
  5. Export constants: `TEST_PROJECT_ID`, `TEST_API_KEY` (the plaintext key for the x-api-key header), `TEST_USER_ID`

IMPORTANT: Do NOT use mocks. The seed inserts real rows into a real PostgreSQL database. The API key must work through Better Auth's real `verifyApiKey` flow.

IMPORTANT: If Better Auth's API key hashing makes direct DB insertion impractical, create the API key through Better Auth's server-side API (`auth.api.createApiKey`) during seed. This requires instantiating the auth instance with the test DB connection.

Replace the existing skipped test stubs with real integration tests that make HTTP requests against the Hono app backed by the seeded test database.

**Test setup within the file:**
- Import the Hono app instance (the full composed app from `apps/api/src/index.ts` or build a minimal one that mounts the entries route with the api-key-auth middleware)
- Import `TEST_API_KEY`, `TEST_PROJECT_ID` from `../test/seed`
- Import `createDb` from `@pleasehold/db` and connect using `process.env.TEST_DATABASE_URL` (set by global setup)
- Use `beforeAll` to create the DB connection and (if needed) create the app instance
- Use `afterAll` to close the DB connection
- Use Hono's `app.request()` method to make requests (no need for a running HTTP server — Hono supports in-process request handling)

**Tests to implement (remove ALL `.skip` calls):**

1. **returns 401 when x-api-key header is missing**
   - `app.request('/api/v1/entries', { method: 'POST', body: JSON.stringify({ email: 'test@example.com' }), headers: { 'Content-Type': 'application/json' } })`
   - Assert: status 401, body.error.code === 'MISSING_API_KEY'

2. **returns 401 when API key is invalid**
   - Same as above but with `x-api-key: 'ph_live_invalid_key_12345'`
   - Assert: status 401, body.error.code === 'INVALID_API_KEY'

3. **returns 400 VALIDATION_ERROR when email is missing**
   - POST with valid API key, body: `{}`
   - Assert: status 400, body.error.code === 'VALIDATION_ERROR', body.error.details[0].field === 'email'

4. **returns 400 VALIDATION_ERROR when unexpected field is sent**
   - POST with valid API key, body: `{ email: 'test@example.com', phone: '555-1234' }`
   - Assert: status 400, body.error.code === 'VALIDATION_ERROR'

5. **returns 201 with entry data including position for valid submission**
   - POST with valid API key, body: `{ email: 'integration-test-1@example.com' }`
   - Assert: status 201, body.data.id exists, body.data.email === 'integration-test-1@example.com', body.data.position >= 1, body.data.createdAt is ISO string

6. **returns 200 with existing entry when same email submitted again (dedup)**
   - POST same email `'integration-test-1@example.com'` again (relies on test 5 having run first — use sequential test ordering or submit a fresh unique email, then submit it again within the same test)
   - Better approach: within this single test, POST `'dedup-test@example.com'` twice
   - First: assert 201
   - Second: assert 200, same `id` and `position` as first

7. **returns 201 and assigns sequential positions**
   - POST `'position-a@example.com'`, note position
   - POST `'position-b@example.com'`, note position
   - Assert second position === first position + 1

8. **returns 201 and stores metadata when provided** (bonus, matches existing stub)
   - POST with `{ email: 'meta-test@example.com', metadata: { source: 'landing' } }`
   - Assert: status 201
   - Query DB directly to confirm metadata was stored: `db.query.entries.findFirst({ where: eq(entries.email, 'meta-test@example.com') })` and check `metadata.source === 'landing'`

IMPORTANT: Use unique email addresses per test to avoid cross-test deduplication conflicts. Include a prefix or UUID in each test's email.
