<objective>
Fix the Docker migration flow so it does not depend on the gitignored `drizzle/` migrations directory. Migrations should be generated from the schema source at build/deploy time, not checked into git.

Purpose: The `drizzle/` directory is gitignored (by design -- migrations are reproducible from schema). But the Docker `migrate` service runs `pnpm db:migrate` which expects pre-generated SQL files in `packages/db/drizzle/`. A clean `docker compose up` from a fresh clone fails because those files don't exist. The fix is to generate migrations before applying them.

Output: Updated docker-compose.yml migrate service that runs `db:generate` before `db:migrate`.
</objective>

  <name>Task 1: Update Docker migrate service to generate migrations before applying</name>
  <files>docker-compose.yml</files>
  <action>
Update the `migrate` service in `docker-compose.yml` to run `db:generate` before `db:migrate`. The command should be a shell invocation that chains both steps:

```yaml
command: sh -c "pnpm db:generate && pnpm db:migrate"
```

This works because:
- The `builder` target stage has all source code including `packages/db/src/schema/*.ts`
- `drizzle-kit` is installed as a devDependency of `@pleasehold/db`
- `pnpm db:generate` runs `turbo db:generate` which runs `drizzle-kit generate` in the db package
- This produces the `packages/db/drizzle/*.sql` files FROM the schema source
- `pnpm db:migrate` then applies those generated SQL files to the database
- `db:generate` does NOT need DATABASE_URL (it only reads schema files and writes SQL)
- `db:migrate` DOES need DATABASE_URL (it connects to PostgreSQL to apply migrations)

Only change the `command` field. Keep all other migrate service config (build, environment, depends_on, networks, restart) exactly as-is.

Preserve the ABOUTME comments at the top of docker-compose.yml.
